Beyond (beyond.tech / docs.beyond.tech) — Deep-Dive Research Brief
Last verified: January 11, 2026

NOTE ON SCOPE & DATA QUALITY
- This report is built from Beyond’s public website, official documentation, app UI pages, audit references, and public GitHub/org pages.
- Several product sections indicate “coming soon”, “mock data mode”, or “maintenance mode”, suggesting the protocol/app is not yet live on mainnet at the time of verification (see “Current Status”).

──────────────────────────────────────────────────────────────────────────────
1) Project Snapshot
──────────────────────────────────────────────────────────────────────────────
Name / Branding
- Name: Beyond (also stylized as ₿eyond)
- Tagline: “Enabling the Bitcoin On-Chain Economy”
- Positioning: “Bitcoin, Now Interoperable” (native Bitcoin L1 interoperability protocol)

Core Idea (1 sentence)
- Beyond is a Bitcoin L1 interoperability protocol that aims to bridge assets to/from Bitcoin L1 and also enable Bitcoin-to-Bitcoin transfers across multiple Bitcoin token standards (“tri-directional” bridging).

Key Claims (from official materials)
- Bridge major assets (e.g., USDT/ETH/SOL) to Bitcoin L1.
- Move BTC, BRC-20s, and Runes to many external chains/L2s.
- Switch between Bitcoin token standards (e.g., BRC-20 ↔ Runes) without leaving Bitcoin L1.
- “80+ chains” and “500+ tokens” are mentioned on the website, while docs also reference 70+ supported chains (exact final list appears planned for post-mainnet).

Core Interop Provider Mentioned
- LayerZero is cited as a key interoperability/messaging layer powering Beyond.

──────────────────────────────────────────────────────────────────────────────
2) The Problem Beyond Targets
──────────────────────────────────────────────────────────────────────────────
Bitcoin Liquidity & Utility Gap
- The project argues that Bitcoin is the largest/most secure base asset but historically lacked the infrastructure to support a full “on-chain economy” (DeFi-style utility), compared to more “composable” ecosystems.

Fragmentation Inside Bitcoin
- Beyond highlights that Bitcoin has many token standards/meta-protocols and layers (L2s), and that liquidity and users are fragmented across these sub-ecosystems.

Cross-Ecosystem Liquidity Fragmentation
- Beyond also frames a broader issue: liquidity is fragmented across many chains and protocols, creating friction for both users (seeking yield/features) and protocols (seeking users/liquidity).

──────────────────────────────────────────────────────────────────────────────
3) What Beyond Actually Provides (Capabilities)
──────────────────────────────────────────────────────────────────────────────
A) Cross-chain (non-Bitcoin → Bitcoin, and Bitcoin → non-Bitcoin)
- “Bi-directional” bridging between Bitcoin L1 and other ecosystems.
- Example intent: bring stablecoins/blue-chips as Bitcoin-native representations, and let Bitcoin-native assets move outward to other chains.

B) Cross-standard (Bitcoin token standards within Bitcoin L1)
- “Cross-standard conversion”: switching between Bitcoin token standards (e.g., BRC-20 ↔ Runes) in a single transaction without leaving Bitcoin L1.
- This is positioned as Beyond’s differentiator beyond typical bridges.

C) “Tri-directional” bridging (as marketed)
- Beyond uses “tri-directional” to mean:
  1) to Bitcoin L1
  2) from Bitcoin L1
  3) within Bitcoin L1 (across standards / Bitcoin layers)

D) Program layer (community / growth mechanisms)
- Leaderboard: a gamified activity ranking.
- Airdrop/Rewards: the app page indicates Season One rewards denominated in $BYD.

──────────────────────────────────────────────────────────────────────────────
4) Architecture & How It Works (as described in docs)
──────────────────────────────────────────────────────────────────────────────
High-level model
- Typical bridge pattern:
  - Lock native tokens on the source chain
  - Mint wrapped tokens on the destination chain
- Beyond describes a mix of on-chain + off-chain components used to coordinate bridging and indexing.

Main components (docs terminology)
1) Node + Indexer
   - Infrastructure that facilitates transactions and indexing across chains, supporting real-time data accuracy/access.
2) Registry
   - A database layer consolidating data from sources (like indexers) and applying extended business logic.
3) Source (Smart Contract, non-Bitcoin chains)
   - Contract that mints/burns tokens on non-Bitcoin-L1 chains.
4) Forge (Bitcoin L1 custody + mint/burn)
   - An MPC / multisig-controlled entity that locks/unlocks tokens on Bitcoin L1.

Forge signer model (important operational detail)
- Docs state: Forge secured by 11 signers with a 7-key threshold for authorization.
- Beyond V1: Forge is permissioned.
- Beyond V2: described as becoming permissionless “via staking”.

User journey examples (docs)
- Avalanche → Bitcoin:
  1) User locks AVAX in Source contract on Avalanche; cross-chain event triggered.
  2) Forge mints wrapped AVAX on Bitcoin; waits 6+ blocks for confirmation.
  3) Indexer/Registry update.
- Bitcoin → Avalanche:
  1) User burns wrapped AVAX via Forge.
  2) After 6+ blocks, event triggers cross-chain message; Indexer/Registry update.
  3) AVAX unlocks in Source contract on Avalanche.
- Bitcoin BRC-20 → Bitcoin Runes:
  1) User locks ORDI (BRC-20) in Bitcoin Forge.
  2) After 6+ blocks, Indexer/Registry update.
  3) Forge mints ORDI as Runes representation and transfers to user.

6) Security Model & Trust Assumptions
──────────────────────────────────────────────────────────────────────────────
Security principles listed in docs
- Encryption (TLS for transit, AES for data at rest).
- Key management with MPC + threshold signing (TSS), splitting control across signers.
- Smart contract security emphasis (audits, pre-deploy review).
- Node security (hardening, monitoring), secure APIs, access controls, intrusion detection, DDoS protection.
- Logging/monitoring and incident response plan.
- Continuous security assessment and penetration testing.

Critical trust points (interpretation of the model)
- Forge custody & signer set is a major trust surface:
  - An MPC/multisig model reduces single points of failure but introduces governance/operational risks (signer integrity, key management procedures, incident response).
- Reliance on cross-chain messaging layer:
  - Beyond publicly ties part of its trust model to LayerZero-based messaging standards and security posture.

Audits
- Beyond’s docs list audits; one public reference is a Zokyo audit report dated Oct 30, 2024 (linked from Beyond docs and listed in Zokyo’s reports index).

Bug bounty
- Docs point to an Immunefi-based bug bounty program, but also state “once Beyond goes live on mainnet” it will launch with a significant, permanent program (i.e., it may not be fully live yet).

10) Due Diligence Checklist (What to verify next)
──────────────────────────────────────────────────────────────────────────────
Technical
- Confirm mainnet launch status and supported chains/standards list once published.
- Inspect contract addresses and verify code (once deployments are posted).
- Validate bridging model for each route (non-BTC ↔ BTC, BTC standard ↔ BTC standard).
- Confirm how LayerZero messaging is configured (endpoints, OFT usage, DVNs/relayers if applicable).

Security
- Read the full audit report(s) and check:
  - scope
  - findings severity
  - remediation status
  - whether audited code matches current deployments
- Confirm bug bounty is live, scope, reward tiers, and response SLAs.

Operations & Governance
- Identify Forge signer set governance, rotation, geographic distribution, incident playbooks.
- Confirm what “permissionless via staking” means in V2 (validators? challengers? slashing conditions?).

Token / Incentives
- Wait for official tokenomics: total supply, allocations, vesting, emissions, governance rights.
- Verify that leaderboard/airdrop rules are finalized and not in “mock” mode.
